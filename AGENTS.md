# OpenBread Embedded Agent Rules

## Scope
- 适用于本仓库全部代码与文档。
- 目标设备为 ESP32-S3，优先满足稳定性、可维护性、低资源占用。
- 默认面向电池供电场景思考实现，除非用户明确要求常驻高性能或常驻联网。

## Hard Requirements
- 代码必须写在职责正确的模块中。
- 代码必须符合嵌入式 C/C++ 风格，简洁、高效、可读。
- 每次改动完成后必须执行 `pio run`。
- 回复必须包含编译结果与资源占用（RAM/Flash）。
- 不允许仅“编译通过”就结束；必须主动检查逻辑正确性与交互正确性。
- UI 不能出现反直觉流程，不能破坏既有按键逻辑。

## Architecture Rules
- `App` 层负责状态编排，不承载具体页面业务细节。
- `ui/pages` 承载页面业务语义与页面私有逻辑。
- `UiManager` 负责输入路由、状态切换、过渡动画、统一渲染编排。
- `bsp` 仅做硬件访问，不放业务判断。
- `services` 承载业务逻辑，不直接操作底层显示绘制细节。

## Embedded Coding Style
- 禁止无必要动态内存分配，优先静态/栈内存与固定容量结构。
- 避免链式嵌套和深层条件嵌套；优先提前返回与小函数拆分。
- 命名清晰、语义直接，避免隐式副作用。
- 复用已有工具函数与布局函数，避免复制粘贴代码块。
- 非必要不新增抽象层，不引入与目标无关的代码。

## Hardware Debug Rules
- 涉及硬件相关代码改动时，必须同时补充足够的串口日志反馈，便于判断当前硬件状态、失败阶段与下一步排查方向。
- 串口日志应聚焦关键状态切换、初始化结果、错误原因与关键测量值，避免无意义刷屏。
- 串口日志必须使用统一前缀，便于过滤和定位，例如 `[SD]`、`[WIFI]`、`[RTC]`、`[ERR]`。
- 调试日志应支持关闭，优先通过宏、配置开关或编译条件控制，避免正式版本长期刷串口影响性能与功耗。

## Hardware Validation Rules
- 涉及硬件驱动、外设初始化、总线通信、供电控制、时序相关改动时，回复中必须区分“已完成编译验证”和“仍需实机验证”的部分。
- 涉及新增或修改硬件初始化逻辑时，必须说明依赖的引脚映射、总线模式、关键假设与限制条件。
- 如果现有命名与实际硬件总线模式不一致，必须主动指出，不允许默认忽略。

## UI And Coordinate Safety
- 优先使用现有布局函数与裁剪逻辑，避免散落硬编码坐标。
- 避免复杂坐标计算；需要计算时拆成中间变量并保证可读性。
- 所有绘制必须考虑边界，防止越界与异常闪烁。
- 交互行为必须保持一致性：
- `Up/Down` 仅负责焦点与翻页。
- `OK` 负责进入/确认。
- `Left` 负责返回上一级。
- 分页、进度条、页码与箭头必须与焦点状态一致。

## Power Guidance Rules
- 除硬性需求外，所有实现默认以“最小资源占用”为目标。
- 优先 O(1) 或低复杂度路径，避免每帧不必要计算。
- 仅在需要时重绘，避免无效渲染。
- 仅在必要时添加状态变量，避免状态膨胀。
- 用户主动触发功能时，才允许初始化和调用相关硬件、无线、外设或高耗能模块。
- 用户未触发相关功能时，应尽可能释放资源、关闭外设、停止后台任务，并保持低功耗状态。
- 功能结束、页面退出、超时失败或无后续任务时，应优先回收相关资源，不允许默认常驻占用。
- 如果 UI 需要配合硬件级别联动逻辑，必须主动向用户报告低功耗方面的可选建议与影响，由用户决定是否采用。
- 如果低功耗策略会影响连接速度、唤醒延迟、扫描能力、动画流畅度、后台保持能力或用户感知行为，必须明确说明副作用。
- 对无线相关功能，默认策略应为“按需开启、任务完成即关闭”，除非用户明确要求常驻连接。

## Change Workflow
- 先定位改动归属模块，再实现。
- 改动后自检：
- 编译是否通过。
- 关键路径逻辑是否与原交互一致。
- 是否引入 UI 显示异常风险。
- 涉及硬件的改动是否具备足够日志辅助排查。
- 是否符合按需启用、及时释放资源的低功耗原则。
- 最终回复必须包含：
- 改了哪些文件。
- 为什么这样改。
- `pio run` 结果。
- RAM/Flash 占用。
- 哪些部分已完成编译验证。
- 哪些部分仍需实机验证。

## Reject Conditions
- 以下情况视为未完成：
- 未编译验证。
- 未汇报 RAM/Flash。
- 代码放错层级。
- 引入反直觉交互。
- 增加明显冗余或低效代码。
- 涉及硬件改动却没有提供足够的串口日志或验证说明。
- 未遵守“用户触发才启用硬件、结束后及时释放资源”的低功耗原则。
