# Open Bread 设备项目大纲（TFT单色反射屏）

## 1. 项目目标
- 目标设备：2.9寸单色反射式 TFT（1-bit，黑白两色，无亮度控制）便携设备。
- 核心功能：设置、音乐、阅读、闹钟、无线遥控。
- 硬件资源：SD 卡、陀螺仪(IMU)、RTC 芯片、马达驱动芯片、DAC。
- 系统要求：高可维护、低耦合、可扩展、非阻塞主循环、UI 风格统一。

## 2. 总体架构
- 架构分层：`App Core -> Services -> BSP Drivers`，UI 独立为表现层。
- 驱动层（BSP）：只负责硬件访问，不包含业务规则。
- 服务层（Services）：封装业务逻辑，向上提供统一接口。
- UI 层：只处理页面、组件、交互反馈，不直接操作底层硬件。
- 应用核心（App）：状态机 + 事件总线 + 调度器，负责系统编排。

## 4. 核心设计原则
- 实时优先：所有主路径满足可预测时延，输入与告警链路优先级最高。
- 非阻塞执行：禁止 `delay()` 与长阻塞 I/O，统一使用 `millis()` 任务调度与状态轮询。
- 工具链约定：构建、烧录与串口监视统一使用 PlatformIO CLI（`pio`）命令，不以 `python -m platformio` 作为默认入口。
- 明确时序预算：为 UI、输入、IMU、存储、音频分别定义周期与最大执行时长预算。
- 中断最小化：ISR 只做置位/入队，不做复杂逻辑、不访问慢速外设、不动态分配内存。
- 内存确定性：关键路径避免堆分配，优先静态内存、固定容量缓冲区与对象池，防止碎片化。
- 功耗可控：支持 `Active/Idle/Sleep` 功耗状态，外设按需上电，空闲时降频或休眠。
- 故障可恢复：外设初始化失败可降级运行（如无 SD 进入只读内建模式），并支持热重试。
- 数据安全：配置写入采用原子更新（临时文件+校验+替换），防止掉电损坏。
- 可观测性：统一日志分级与事件追踪，关键故障保留最小诊断信息（重启原因、错误码、时间戳）。
- 硬件抽象隔离：业务与 UI 不直接依赖芯片寄存器，全部经 BSP 接口访问，便于换型与测试。
- 接口稳定：跨层通信使用小而稳定的数据结构与事件协议，减少模块间隐式耦合。
- 可测试可量产：关键服务具备仿真或回放入口，启动自检覆盖显示、存储、IMU、RTC、马达、DAC。

## 5. App Core 设计
### 5.1 状态机（AppState）
- `Boot`：上电初始化。
- `Home`：主菜单与导航。
- `Reader`：阅读功能页。
- `Music`：音乐功能页。
- `Alarm`：闹钟功能页。
- `Remote`：无线遥控页。
- `Settings`：设置页。
- `Sleep`：低功耗待机。
- `Error`：错误处理与恢复。

### 5.2 事件总线（EventBus）
- 事件类型：`Input`、`Timer`、`RtcAlarm`、`ImuGesture`、`SdMounted`、`AudioState`、`Navigation`。
- 事件载荷：固定小结构体（type + 参数），避免堆内存分配。
- 分发策略：主循环内轮询分发，确保线程模型简单稳定。

### 5.3 任务调度（Scheduler）
- 输入扫描：`10ms`。
- IMU 轮询：`20ms`。
- UI tick：`33ms`（约 30fps 逻辑刷新）。
- 闹钟检查：`1000ms`。
- 存储刷写：后台低频（避免频繁写入）。

## 6. UI 与渲染框架（单色屏专项）
### 6.1 本章总结
- UI 采用三层结构：`主菜单 -> 子菜单 -> 详情页`。
- 主菜单采用纵向滚动列表，中心固定矩形焦点框，菜单内容上下移动。
- 强制使用单色屏友好策略：1bpp、脏区刷新、低闪烁、高对比。
- 主菜单到子菜单必须使用统一过渡动画，形成稳定的产品识别感。

### 6.2 信息架构
- 主菜单（Home）：`Music / Reader / Alarm / Remote / Settings` 五个入口，负责功能选择。
- 子菜单（Section）：模块内功能分组与操作列表。
- 详情页（Detail）：单对象或单任务的具体操作界面。
- 导航约束：任何详情页最多两次返回到主菜单，路径必须确定且一致。

### 6.3 主菜单交互逻辑（纵向列表）
- 入口顺序默认：`Music -> Reader -> Alarm -> Remote -> Settings`。
- 中心固定焦点框：使用完整矩形描边，焦点框本身不移动。
- 菜单项滚动：`Up/Down` 输入触发列表整体 Y 方向平移，焦点框不动。
- 焦点判定：最终停在中心的菜单项即当前焦点项。
- 循环策略：默认循环滚动，首尾无缝衔接。
- 快速输入策略：动画可被新输入打断并续动，避免排队卡顿。

- 当前代码与本节规范一致：焦点框为完整矩形描边样式。
- 循环滚动与“动画可打断续动”已实现。

### 6.4 主菜单状态机
- `Idle`：等待输入，当前焦点稳定。
- `Sliding`：列表上下滚动中。
- `TransitionToSection`：主菜单到子菜单过渡动画中。
- 状态迁移：
  - `Idle + Up/Down -> Sliding`
  - `Sliding + 动画完成 -> Idle`
  - `Idle + Confirm -> TransitionToSection`
  - `TransitionToSection + 动画完成 -> 子菜单激活`

- 状态迁移方向与本节规范一致：`Idle + Up/Down -> Sliding`。
- `Idle + Confirm -> TransitionToSection` 与转场完成进入 `Section` 已实现。
- 转场期间通过状态隔离实现输入锁定（仅转场计时，不处理主页导航输入）。

### 6.5 主菜单渲染规范
- 图层顺序：
  - 背景层（简洁纹理或几何分区，低复杂度）。
  - 菜单项层（中心项高权重，侧项弱化）。
  - 焦点框层（完整矩形描边，始终最上层）。
- 菜单项样式：
  - 中心项：图标与标题使用最高对比与最大权重。
  - 侧项：缩小或降低描边权重，避免与中心项抢视觉。
- 文本规则：
  - 标题长度超过阈值时可局部滚动显示，但不影响主滚动动画。

### 6.6 主菜单到子菜单过渡动画（重点）
- 目标原则：先主菜单退场，再子菜单入场，形成明确“离场 -> 进场”节奏。
- 触发条件：主菜单 `Idle` 状态下收到 `Confirm`。
- 动画总时长：建议默认 `420ms`。
- 动画分段（正向）：
  - `0%~29%`（约 0~120ms）：主菜单背景图整体向左滑出屏幕。
  - `14%~52%`（约 60~220ms）：主菜单三个可见项（图标+标题）按“上 -> 中 -> 下”依次向左滑出并消失。
  - `52%~100%`（约 220~420ms）：子菜单整体从屏幕下方向上滑入并落位（`y: +height -> 0`）。
- 动画曲线：
  - 退场段（背景与菜单项）使用 `ease-in`，强化起势离场感。
  - 入场段（子菜单）使用 `ease-out`，确保落位稳定。
- 图层与运动约束：
  - 背景层与主菜单项层仅做 X 轴位移；子菜单层仅做 Y 轴位移。
  - 图标与标题必须同位移、同曲线，不允许分离运动。
  - 过渡中不依赖透明度、灰度或复杂混合，仅使用 1bpp 可实现效果。
- 状态机约束：
  - 进入过渡态：`Home + Confirm -> TransitionToSection`。
  - 过渡完成：`TransitionToSection -> Section`。
  - 过渡期间输入锁定，防止重复触发与状态竞争。
- 落位与一致性：
  - 子菜单落位后再解锁输入，并激活子菜单默认焦点项。
  - 过渡结束后各层偏移量归零，避免残留位移影响后续绘制。
- 验收标准：
  - 视觉上可明确感知“背景先退、菜单项后退、子菜单再入场”。
  - 菜单项依次退场节奏清晰，不出现整块同时消失。
  - 无闪屏、无跳点、无明显残影。

### 6.6.1 子菜单返回主菜单过渡动画（倒序）
- 目标原则：返回动画严格遵循“正向转场倒序”，形成一致的视觉闭环。
- 触发条件：
  - 在 `Section` 页按 `LEFT/Back`，触发返回主菜单转场。
  - 在 `Detail` 页按 `LEFT/Back`，先返回 `Section`，再由 `Section` 触发返回主菜单。
- 动画总时长：建议默认 `420ms`（与正向复杂转场保持一致节奏）。
- 动画分段（倒序）：
  - `0%~48%`（约 0~200ms）：子菜单整体向下滑出屏幕（`y: 0 -> +height`，`ease-in`）。
  - `43%~81%`（约 180~340ms）：主菜单三个可见项（图标+标题）按“下 -> 中 -> 上”依次从左侧滑入。
  - `71%~100%`（约 300~420ms）：主菜单背景图从左侧滑入并最终落位。
- 图层与运动约束：
  - 子菜单层仅做 Y 轴位移；主菜单项与背景层仅做 X 轴位移。
  - 图标与标题必须同位移、同曲线，不允许分离运动。
  - 过渡期间保持 1bpp 高对比表现，不依赖透明混合与灰度特效。
- 状态机约束：
  - 新增/使用返回过渡态：`ToHomeTransition`。
  - 状态迁移：`Section + Back -> ToHomeTransition -> Home`。
  - 过渡期间输入锁定，防止重复触发与状态竞争。
- 落位与一致性：
  - 回到主菜单后保留离开前的焦点索引（`focusIndex`）以保持导航连续性。
  - 所有转场偏移量归零后再解锁输入。
- 验收标准：
  - 视觉上能明确感知“先子菜单离场，再主菜单项入场，最后背景归位”。
  - 无闪屏、无跳点、无明显残影。

### 6.7 视觉语言（单色反射屏）
- 仅使用黑白两色，通过反白与线条粗细表达层级。
- 统一图标网格、描边粗细、圆角半径与间距系统。
- 强对比优先，可读性优先于装饰性。
- 动效不依赖灰阶、透明和复杂混合效果。

### 6.8 渲染实现与性能预算
- 渲染模型：`1bpp framebuffer` + `dirty-rect` 局部刷新。
- UI tick：33~50ms（20~30fps 逻辑刷新）。
- 单帧预算：目标 <= 8ms（不含慢速 I/O）。
- 主菜单滚动与转场动画期间禁止触发文件系统读取与重型排版。
- 丢帧优先级：先保证焦点落位正确，再补绘侧项细节。

### 6.9 UI 模块分工
- `UiManager`：路由、页面栈、过渡状态机、输入锁定。
- `HomePage`：主菜单滚动状态机、焦点索引、过渡触发。
- `Render1bpp`：脏区收集、图层绘制、提交刷新。
- `ThemeMono`：角标尺寸、间距、线宽、动画时长、曲线参数。
- `pages/*`：子菜单与详情页业务展示，不直接操作底层硬件。

### 6.10 验收标准
- 主菜单纵向滚动手感稳定，中心焦点框始终静止且识别清晰。
- 主菜单到子菜单过渡动画连续、无错位、无明显闪屏。
- 三层导航逻辑一致，返回路径可预测。
- 在目标帧率下可长时间运行，无明显卡顿与抖动。

## 7. 功能模块职责
### 7.1 SettingsService
- 管理系统设置：音量、震动强度、时间制式、主题参数、遥控配置。
- 持久化策略：写入 NVS 或文件系统配置文件。

### 7.2 MusicService
- 管理音乐播放状态（播放/暂停/下一曲/进度）。
- 读取音源索引（SD 卡目录/播放列表）。
- 调用 DAC 输出路径并对接 UI 状态更新。

### 7.3 ReaderService
- 文本读取、分页、书签定位。
- 字号与行高配置，适配单色屏阅读密度。
- SD 卡资源读取与缓存窗口管理。

### 7.4 AlarmService
- 闹钟计划表管理（单次/重复/贪睡）。
- RTC 对时与闹钟触发对接。
- 响铃联动：DAC 播放 + 马达触觉反馈。

### 7.5 RemoteService
- 无线遥控命令封装与发送。
- 遥控页 UI 只发语义命令，不耦合具体协议细节。

### 7.6 GestureService
- IMU 数据采样与手势识别（翻页、甩动、抬腕等）。
- 识别结果发布为统一事件供 UI/服务订阅。

### 7.7 HapticService
- 马达震动模式（短促、双击、长振、渐进）。
- 场景联动：按键反馈、闹钟提醒、错误提示。

## 8. BSP 驱动边界
- DisplayMonoTft：初始化、缓冲区提交、局部刷新。
- SdCardDriver：挂载、文件读写、容量与错误状态。
- ImuDriver：采样、姿态/加速度原始数据提供。
- RtcDriver：读写时间、闹钟中断配置。
- MotorDriver：震动触发与时长控制。
- DacDriver：输出增益、音频路径控制。

## 9. 数据与存储
- `data/models`：领域模型（闹钟项、书签项、歌曲元信息、设备设置）。
- `data/storage`：存储仓库接口与实现（配置、书签、历史记录）。
- 文件组织建议：
  - `/config/settings.json`
  - `/reader/bookmarks.json`
  - `/music/library.idx`
  - `/logs/system.log`

## 10. 错误处理与恢复
- 启动阶段记录硬件初始化结果；失败进入 `Error` 状态并给出 UI 提示。
- SD/RTC/IMU 允许热重试，不阻塞主循环。
- 异常统一上报到事件总线，按模块可恢复策略处理。

## 11. 性能与资源约束
- RAM 优先级：UI framebuffer > I/O 缓冲 > 业务缓存。
- 避免动态内存碎片：关键路径使用静态或对象池。
- 日志分级：`ERROR/WARN/INFO/DEBUG`，发布版本默认关闭 DEBUG。
- UI 刷新预算：优先保证输入响应，再追求动画完整性。

## 12. 测试与验证范围
- 启动自检：Display/SD/IMU/RTC/Motor/DAC 全链路探测。
- 服务验证：每个 Service 提供最小行为测试入口。
- 场景测试：阅读连续翻页、闹钟触发、遥控指令连发、长时间待机唤醒。
- 回归重点：状态切换、事件洪峰、SD 拔插、RTC 异常时间。

## 13. 里程碑定义
- M1：架构骨架可编译，主循环+状态机+事件总线跑通。
- M2：UI 首页 + 设置页 + 基础输入完整联通。
- M3：阅读模块（SD 读取 + 分页 + 书签）可用。
- M4：音乐模块（文件索引 + DAC 控制）可用。
